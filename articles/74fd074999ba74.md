# はじめに

直近、業務で Flutter アプリにカレンダーへの日程登録機能を実装する機会がありました

Plugin の技術選定をしていく中で、「AI エージェントが使える今の時代って、もう少し選択肢を広げて考えられるんじゃないか？」とふと思いました

開発の敷居が下がりつつある今だからこそ、技術選定の在り方を改めて考えたいと思い執筆しています

:::message
**注意**
特定のパッケージや Plugin を否定したいわけではないです
要件が合えば地道に作る必要はなくありがたいと思います！
「パッケージが提供できる機能」と「プロジェクトとして解決したい技術的課題」は別の話だと思っています
:::

# Plugin の選定をした話

カレンダー登録機能の実装にあたり、まずは既存のパッケージを探すところからはじめました

比較した観点はざっくりこんな感じです

- メンテナンスは活発か？
- ユーザー体験を考慮した登録のしやすさはどうか(≒要件にあうか)
- 実装工数・QA コスト（パーミッション対応含む）は下げられるか

# 選定の判断基準

ユーザー体験を考慮すると、シームレスに他カレンダーアプリへの日程を登録できることが望ましいと考えていました

よくある実装だと、カレンダーの読み書きパーミッションをアプリ側でリクエストして、許可をもらったうえでイベントを書き込む形になります

ただこのやり方、パーミッションの説明文（usage description）を用意したり、許可・拒否それぞれのケースを QA でカバーしたりと工数がかかります

調べた結果、[eventide](https://pub.dev/packages/eventide) パッケージがこの要件を満たしていることがわかりました

eventide が良かったのは、OS のネイティブな仕組みをうまく活用している点です

- **iOS：EventKitUI** を使って OS 標準のカレンダー登録 UI をモーダルで開く
- **Android：Intent** で端末の標準カレンダーアプリを呼び出す

どちらもアプリ側でパーミッションを要求する必要がなく、パーミッション対応をまるごと省けます

[device_calendar_plus](https://pub.dev/packages/device_calendar_plus) は EventKitUI の新規登録には対応していませんでした
他にもいくつか見てみましたがメンテが止まっているものもあり、消去法的に eventide として選定しました

# バイナリアップロード時にAppleのチェックで引っかかった

選定を終えて実装も完了し、App Store Connect にバイナリをアップロードしようとしたところ、バイナリレベルでアップロード自体が弾かれてしまいました

届いたエラーがこちらです

```
Please correct the following issues and upload a new binary to App Store Connect.
ITMS-90683: Missing purpose string in Info.plist - Your app's code references one or more APIs
that access sensitive user data, or the app has one or more entitlements that permit such access.
The Info.plist file for the "Runner.app" bundle should contain a NSCalendarsUsageDescription key
with a user-facing purpose string explaining clearly and completely why your app needs the data.
If you're using external libraries or SDKs, they may reference APIs that require a purpose string.
While your app might not use these APIs, a purpose string is still required.
For details, visit: https://developer.apple.com/documentation/uikit/protecting_the_user_s_privacy/requesting_access_to_protected_resources.
Apple Developer Relations
```

原因を調べると、eventide の内部実装でレガシー API である `requestAccess(to:completion:)` を使っている箇所がありました

https://developer.apple.com/documentation/eventkit/ekeventstore/requestaccess(to:completion:)

https://github.com/sncf-connect-tech/eventide/blob/07099c7cc88d8598e1546e4b439dcb5cfb64c5d9/ios/eventide/Sources/eventide/PermissionHandler/PermissionHandler.swift#L62

このレガシー API がバイナリに含まれているため、チェックに引っかかり `NSCalendarsUsageDescription` が必要と機械的に検知されました

選定の段階でここまで確認できていなかったのは反省点でした

対処としては、サポート下限が iOS 18 のプロジェクトではあるものの、一時的に `NSCalendarsUsageDescription` の文言を追加して回避しました

# AI時代における「自作」という選択肢

ただ、AI エージェントが使える今の時代において、実装コストの敷居は変わってきていると感じます

「ネイティブ（Swift/Kotlin）のコードを書くのは大変だからパッケージに任せよう」という判断もできますが、AI エージェントに指示を出せば、ネイティブの実装もある程度の精度で書いてもらえます

試しに Claude Code のプランモードで以下のような指示を出し、設計から実装まで行いました

```
新しいサンプルを作ってください

作りたいサンプルはeventideと自作のメソッドチャンネルを作る比較機能です

・https://pub.dev/packages/eventide パッケージを使って、createEventThroughNativePlatformを呼んで開くこと
・createEventThroughNativePlatformを調査して同等のメソッドチャンネルを用いて自作すること(pigionも使って型安全にすること)
・dart側とネィティブコード側でそれぞれテストコードも導入すること 
・1ページにボタンを2つのボタンを設けること
```

- **iOS：** `EKEventEditViewController`（EventKitUI）を Swift で直接呼び出す実装
- **Android：** ICS ファイルを生成して Intent ベースで標準カレンダーアプリを開く Kotlin 実装

サンプルをこちらに置いています（Eventide との カスタム Pigeon の比較もできます）

https://github.com/kentomiya89/flutter_playground

# 技術選定はチームの体制も踏まえて考える

ただ、自分が携わっているプロジェクトでは Swift/Kotlin のネイティブコードにある程度の知見を持って継続的にメンテできる体制にはなっていません

自作するということは、将来的に

- AI はあれど、OS API（Swift/Kotlin）をある程度キャッチアップしてメンテしたり、AI に設計をどうハンドリングさせるかを判断できる人が必要になる
- Flutter のバージョンアップに伴う追従コストが発生する

という話にもなります

そのあたりを踏まえると、今回は自作の選択は見送りました

ただ、チームの体制が変わったり、ネイティブエンジニアが増えたりしたタイミングで改めてディスカッションする余地はあると思っています

# まとめ

AI エージェントによって技術の敷居が下がり、以前は手間だった実装が現実的な選択肢になってきました

だからこそ「技術的に実装できるか？」だけでなく

- 要件をちゃんと満たせるか？
- 組織・チームとして継続的にメンテできるか？
- 実装コストとメンテコストのバランスはとれているか？

といった観点を含めた技術選定がより重要になってきていると感じています

取れる選択肢が増えた分、色々な要因を考慮しながら技術選定していきたいと思っています
